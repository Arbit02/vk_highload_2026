## Определение 
Google Play - это глобальный маркетплейс, где Google выступает посредником между разработчиками контента и пользователями Android.

## Аудитория Google Play
- MAU - 2.5 млрд.
- DAU -  675 млн.
DAU расчитывалась, как 27% пользователей(процент вычислялся на основе данных из Steam).

## MVP функционал
- Авторизация и аутентификация пользователя
- Витрина
- Страница приложения
- Поиск
- Загрузка APK-файлов
- Управление установками (Пользователь может видеть все скачанные им с сервиса приложения, отслеживать их прогресс установки/обновления)
- Комментарии к продуктам
- Рейтинг продукта на основе комментариев и оценок пользователей

## Ключевые продуктовые решения
- Поддержка возобновления загрузки при обрыве связи
- Все загружаемые файлы на сервер должны проверяться на безопасность

## Схемы балансировки
**Geo-based DNS:**
Технология маршрутизации запросов пользователей к ближайшему или наиболее подходящему серверу на основе их географического положения

**BGP Anycast**
 Метод маршрутизации, при котором один IP-адрес назначается нескольким серверам

## Механизм балансировки трафика
**Active Health Checks:**
Постоянно выполняются проверки доступности edge-узлов, CDN-пулов и origin-серверов. При обнаружении деградации или отказа узел автоматически исключается из маршрутизации DNS и Anycast.

**Client-side Retry Logic:**
Если при загрузке приложения возникает ошибка или сервер не отвечает, клиентское приложение автоматически повторяет запрос и пытается подключиться к другому серверу или CDN-узлу.


# Локальная балансировка нарузки
## Схема балансировки нагрузки
Реализуем в ДЦ схему балансировку, состоящую из 2 уровней
### L4
- **Технология:** Linux Virtual Server [LVS]
- **Принцип работы:** Virtual Server via Direct Routing (Входящий трафик направляется на балансировщик, который распределяет его между узлами L7, 
а исходящий трафик направляется прямо к клиенту)
- **Резервирование:** Схема "2N": Keepalived управляет плавающим IP-адресом, автоматически перемещая его на резервный сервер в случае сбоя основного.

### L7
- **Технология:** Nginx
- **Функции:** SSL Termination.
- **Минимизация нагрузки:** Session tickets
- **Резервирование:** Схема "N+1": В случае сбоя на 1 из балансировщик нагрузка плавно распределится по оставшимся

## Рассчёт количества балансировщиков
Выполним расчет расчет на основе наиболее загруженного дата-центра - США: DC1. Показатели возьмем из предыдущего раздела
- **Пиковый RPS**: 207 865
- **Пиковый трафик:** 208 Гбит/c

### Узлы L4
- **Целевая конфигурация:** Серверы, поддерживающие подключение к сети на скорости 100 Гбит/с.
- **Ограничитель:** пропускная способность канала
- **Активные узлы:** 208 / 100 = 2.08 = 3 серверов

Т.к мы используем схему 2N: берем в 2 раза больше, итого **6 серверов**

### Узлы L7
- **Целевая конфигурация:** Серверы, поддерживающие подключение к сети на скорости 100 Гбит/с.
- **Ограничитель:** пропускная способность канала; SSL Termination
- **пропускная способность канала** : нужно 3 сервера
 **SSL-Termination**
207 865 * 0.5 = 103 933 CPS
Сервер с 16 CPU обеспечивает производительность 6 676 CPS
103 933 / 6 676 = 16 серверов

Мы используем схему N+1 соответственно нам необходимо **17 сервера**

**Итого**
- **L4:**  6 серверов
- **L7:** 7 серверов

## Индексы
Размеры индексов в оперативной памяти (RAM) вычисляется по формуле: Кол-во записей * (Сумма размеров полей индекса + 8 байт (указатель, который связывает запись в индексе с конкретными данными на диске или в другой области RAM)) * 1.25 (overhead B-Tree).
Коэффициент 1.25 - коэффициент запаса для будущих вставок
| Таблица | Поле | Тип индекса | Комментарий | Размер индекса в RAM |
|---------|------|-------------|-------------|-------------|
| users | username | B-Tree | Используется при каждой авторизации для поиска пользователя по логину| 2.5 млрд. * (50 + 8) * 1.25 = 21.1 ГБ |
| apps | owner_id | B-Tree | Позволяет быстро получить все приложения, созданные конкретным разработчиком.| 3.5 млн. * (8 + 8) * 1.25 =  = 8.3 МБ|
| apps | (category_id, created_at) | Composite B-Tree | Главная страница — топ новых в категории | 3.5 млн. * (8 + 8 + 8) * 1.25 = 12 МБ |
| library | user_id | Partition Key (ScyllaDB) | 17.5 млрд строк, все запросы по user_id | 17.5 млрд. * (8 + 8) * 1.25 = 40.7 ГБ |
| media | (app_id, is_icon) | Partial B-Tree | Иконка нужна всегда при показе аппки | 17.5 млн. * (8 + 2 + 8) * 1.25 = 46.9 МБ |
| reviews | app_id | B-Tree | Страница приложения — все отзывы | 11 млн. * (8 + 8) * 1.25 = 26.2 МБ |
| reviews | (app_id, created_at) | Composite B-Tree | Сортировка отзывов по дате | 11 млн. * (8 + 8 + 8) * 1.25 = 39.3 МБ |
| sessions | token | Redis Key с TTL | Проверка авторизации при каждом запросе | 6.7 млн. * (255 + 8) * 1.25 = 262 МБ |
| sessions | user_id | Redis Set | Поиск всех сессий пользователя (для логаута)| 6.7 млн. * (8 + 8) * 1.25 = 15.9 МБ |

### Резервирование 
| СУБД | Схема | Количество | Обоснование |
|------|-------|------------|-------------|
| PostgreSQL | Один главный, два копии | 1 Master + 2 Replica | Главный для записи, копии для чтения. Если главный упал — копия становится главной |
| Cassandra | Данные копируются на 3 узла | 3 | Любой узел может упасть — данные не потеряются |
| Redis | Один главный, два наблюдателя | 1 Master + 2 Sentinel | Наблюдатели следят за главным. Если главный упал — выбирают нового |
| MinIO | Данные копируются или кодируются | 3+ узла | Файлы не должны потеряться при поломке диска |

## Балансировка запросов / мультиплексирование подключений
| СУБД / Хранилище | Механизм | Суть работы |
|-----------------|----------|--------------|
| PostgreSQL | PgBouncer + реплики для чтения | Приложения подключаются к PgBouncer, который держит пул соединений с мастером и репликами. Запросы на запись идут на мастер, на чтение — на реплики (round-robin) |
| Cassandra | Token-aware routing | Драйвер вычисляет по user_id (ключ шардирования library), на каком узле лежат данные, и отправляет запрос туда |
| Redis | Redis Cluster + хэш-слоты | Ключ session:{token} хэшируется в слот. Клиент знает, какой мастер отвечает за этот слот, и идёт прямо к нему. При переключении мастера клиент обновляет топологию |
| MinIO | Geo-DNS + CDN | Иконки и скриншоты приложений раздаются из MinIO кластера с несколькими регионами. Пользователь получает файл из ближайшего региона, а тяжёлые файлы -  дополнительно кешируются в CDN |


## Схема резервного копирования
| Хранилище | Что бэкапим | Как бэкапим | Комментарий |
|-----------|-------------|--------------|-------|
| PostgreSQL | users, apps, reviews, media (метаданные), categories | Ежедневный полный бэкап | Можно восстановить на любой момент времени. Например, если случайно удалили отзывы или приложение |
| Cassandra | library | Ежедневные снапшоты через Cassandra Manager| Полный дамп каждый час нецелесообразен |
| Redis | sessions | AOF (каждую секунду) + RDB (раз в час) | При падении Redis теряем максимум 1 секунду сессий |
| MinIO | Иконки и скриншоты приложений | Георепликация между регионами | Устойчивость за счёт копирования в другой регион. При падении одного дата-центра файлы будем брать из другого |
