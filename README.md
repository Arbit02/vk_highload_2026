## Определение 
Google Play - это глобальный маркетплейс, где Google выступает посредником между разработчиками контента и пользователями Android.

## Аудитория Google Play
- MAU - 2.5 млрд.
- DAU -  675 млн.
DAU расчитывалась, как 27% пользователей(процент вычислялся на основе данных из Steam).

## MVP функционал
- Авторизация и аутентификация пользователя
- Витрина
- Страница приложения
- Поиск
- Загрузка APK-файлов
- Управление установками (Пользователь может видеть все скачанные им с сервиса приложения, отслеживать их прогресс установки/обновления)
- Комментарии к продуктам
- Рейтинг продукта на основе комментариев и оценок пользователей

## Ключевые продуктовые решения
- Поддержка возобновления загрузки при обрыве связи
- Все загружаемые файлы на сервер должны проверяться на безопасность

## Схемы балансировки
**Latency-based DNS:**
DNS-резолвер пользователя направляется на ближайшую точку присутствия на основе геолокации и сетевой задержки. Используется короткий TTL (60–120 сек), что позволяет быстро перенаправлять пользователей на другие регионы при сбоях или перегрузке дата-центров.

## Механизм балансировки трафика
**Active Health Checks:**
Постоянно выполняются проверки доступности edge-узлов, CDN-пулов и origin-серверов. При обнаружении деградации или отказа узел автоматически исключается из маршрутизации DNS и Anycast.

**Client-side Retry Logic:**
Если при загрузке приложения возникает ошибка или сервер не отвечает, клиентское приложение автоматически повторяет запрос и пытается подключиться к другому серверу или CDN-узлу.


# Локальная балансировка нарузки
## Схема балансировки нагрузки
Реализуем в ДЦ схему балансировку, состоящую из 2 уровней
### L4
- **Технология:** Linux Virtual Server [LVS]
- **Принцип работы:** Virtual Server via Direct Routing (Входящий трафик направляется на балансировщик, который распределяет его между узлами L7, 
а исходящий трафик направляется прямо к клиенту)
- **Резервирование:** Схема "2N": Keepalived управляет плавающим IP-адресом, автоматически перемещая его на резервный сервер в случае сбоя основного.

### L7
- **Технология:** Nginx
- **Функции:** SSL Termination.
- **Минимизация нагрузки:** Session tickets
- **Резервирование:** Схема "N+1": В случае сбоя на 1 из балансировщик нагрузка плавно распределится по оставшимся

## Рассчёт количества балансировщиков
Выполним расчет расчет на основе наиболее загруженного дата-центра - США: DC1. Показатели возьмем из предыдущего раздела
- **Пиковый RPS**: 207 865
- **Пиковый трафик:** 208 Гбит/c

### Узлы L4
- **Целевая конфигурация:** Серверы, поддерживающие подключение к сети на скорости 100 Гбит/с.
- **Ограничитель:** пропускная способность канала
- **Активные узлы:** 208 / 100 = 2.08 = 3 серверов

Т.к мы используем схему 2N: берем в 2 раза больше, итого **6 серверов**

### Узлы L7
- **Целевая конфигурация:** Серверы, поддерживающие подключение к сети на скорости 100 Гбит/с.
- **Ограничитель:** пропускная способность канала; SSL Termination
- **пропускная способность канала** : нужно 3 сервера
 **SSL-Termination**
207 865 * 0.5 = 103 933 CPS
Сервер с 16 CPU обеспечивает производительность 6 676 CPS
103 933 / 6 676 = 16 серверов

Мы используем схему N+1 соответственно нам необходимо **22 сервера**

**Итого**
- **L4:**  6 серверов
- **L7:** 7 серверов
